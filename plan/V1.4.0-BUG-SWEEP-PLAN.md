## Bucket B — Fix this release (11 cards)

### B1. `9919627058` — Duplicate activities on challenge completion (xs)

**RCA:** `ChallengeEngine::complete_challenge()` calls `do_action('wb_gam_challenge_completed', ...)` twice (lines 265 + 275, identical). Every listener — ChallengeStream (BP activity insert), WebhookDispatcher, NotificationBridge, TransactionalEmailEngine — runs twice. Visible as duplicate BP activity rows, also doubles webhook deliveries + emails.

**Fix:** Remove the duplicate `do_action` line. Keep the one at the end of the success path (after counters/totals are persisted, before return). Add a regression test that hooks the action and asserts call count = 1 per completion.

**Impact:** Touches one file. Existing duplicate-prevention idempotency keys in ChallengeStream may already mask this for the BP activity row — verify by checking whether `bp_activity_add` has a uniqueness guard or not.

**Verification:** Complete a challenge with WP_DEBUG_LOG. Hook the action with a counter. Expect count = 1.

### B2. `9919642373` — BP activity filter label collision (xs)

**RCA:** `ActivityIntegration::register_activity_types()` (lines 97-115) registers 4 activity types (`badge_earned`, `level_changed`, `kudos_given`, `challenge_completed`) all with the same context-group label `__('Gamification', 'wb-gamification')` passed as the 5th arg to `bp_activity_set_action`. BP groups filter dropdowns by that string — all four collapse into a single "Gamification" filter row.

**Fix:** Either (a) keep one group label but distinct sub-labels per type (preferred — matches BP UX), or (b) introduce a filter `wb_gam_activity_context_label` so sites can override. Go with (a) — set the 5th arg to per-type human label: "Badge earned", "Level up", "Kudos sent", "Challenge complete".

**Impact:** One file (`ActivityIntegration.php`). String change — translation file regenerates.

**Verification:** Activity directory filter dropdown shows 4 distinct entries instead of one collapsed "Gamification".

### B5. `9863259749` — IA toast empty-text bug (s)

**RCA:** `NotificationBridge::on_level_changed` (lines 168-175) pushes `{type:'level_up', levelName, iconUrl}` — no `message` key. `on_streak_milestone` (lines 184-192) pushes `{type:'streak_milestone', days}` — same. `toast.js:91` reads `toast.message || ''`, so both render empty `<strong>` text.

**Fix:** Backfill `message` in both push points. Use translated strings: `sprintf(__('You reached level %s', 'wb-gamification'), $levelName)` and `sprintf(_n('%d-day streak!', '%d-day streak!', $days, 'wb-gamification'), $days)`. Keep the typed fields (`levelName`, `iconUrl`, `days`) for richer rendering once toast.js learns to consume them — current change is purely additive.

**Impact:** Two methods in `NotificationBridge.php`. No DB change. Adds two i18n strings — regen .pot.

**Verification:** Trip a level-up by awarding enough points. Toast shows "You reached level Champion" instead of empty.

### B6. `9920003216` — Challenge time saved/rendered mismatch (s)

**RCA:** Three-way timezone mismatch:
- `admin-rest-form.js:99-110` converts datetime-local → UTC for save.
- `BadgeEngine` (lines 186, 221, 317, 346, 452) compares stored strings via `gmdate()` (UTC vs UTC) → consistent.
- `ChallengeEngine.php:101-102, 302-303` compares same stored strings via MySQL `NOW()` which returns server-local time → UTC-offset mismatch. Challenges become "active" early/late by the server offset.
- Admin form re-reads the UTC string and interprets it as site-local on render → displayed time drifts further from saved value.

**Fix:** Two-part.
1. **DB query side:** Replace `NOW()` with `UTC_TIMESTAMP()` in ChallengeEngine WHERE clauses (lines 101-102, 302-303). This aligns with the UTC-stored columns and matches BadgeEngine's `gmdate()` pattern.
2. **Admin form render side:** When loading datetime-local from a UTC `Y-m-d H:i:s` string, convert UTC → local in JS (`new Date(`${utc} UTC`).toISOString().slice(0,16)`) before populating the input. This is in `admin-rest-form.js` field-init.

**Impact:** 2 SQL queries + 1 JS helper. Touches challenges + community-challenges queries (same engine). Add unit test for ChallengeEngine::get_active_for_user with both UTC and pre-UTC server timezones.

**Verification:** On a site in `wp_timezone_string = America/Los_Angeles` (UTC-8), create a challenge starting "today at 9 AM". Confirm: (a) admin form re-renders 9 AM after save+reload, (b) challenge becomes active at 9 AM PT, not 1 AM PT or 5 PM PT.

### B7. `9919339320` — Admin notices inside settings container (s)

**RCA:** `SettingsPage.php:529` uses `<div class="wrap wbgam-wrap">`. WordPress targets `.wrap` for `admin_notices` injection. The plugin's custom `<header>`, sidebar, content layout starts inside that wrap, so notices appear between `<div class="wrap">` and `<header class="wbgam-page-header">` — visually trapped inside the plugin chrome and inheriting `wbgam-wrap` padding.

**Fix:** Move `wbgam-wrap` styling onto a nested `<div class="wbgam-shell">` and let `<div class="wrap">` stay bare. Notices then render in the standard WP location (above the page header), unstyled by the plugin. Update CSS selectors that were targeting `.wrap.wbgam-wrap` → `.wbgam-shell`.

**Impact:** Settings template + ~6 CSS selectors. Visual regression risk on every settings page — verify each tab renders identically post-refactor.

**Verification:** Trigger an admin notice (e.g. save settings successfully). Notice appears above the Wbcom-styled header, not inside it.

### B8. `9919566015` — Delay in points updates (m)

**RCA:** Three compounding delays:
1. `Registry.php:181` — every action with `async=true` OR `repeatable=true` is dispatched via Action Scheduler (`Engine::process_async`). AS runs at its own tick interval (default 1 min).
2. `toast.js:133` — `setInterval(checkToasts, 30000)` polls every 30s.
3. `member-points` and hub-tile blocks are server-rendered with no client refresh on award/skipped events. Members see stale totals until the next page navigation.

**Fix:** Three-layer.
1. **Manifest re-audit:** Most BP/WC actions don't need `async`. Review the manifest, drop `async=true` from low-volume actions (kudos, comment, profile update). Keep async for high-volume / bursty actions (mass imports, batch awards). Document the policy in CLAUDE.md.
2. **Toast poll interval:** Drop from 30s → 10s when a known award is pending (use sessionStorage flag set by the originating REST call). Plain idle polling stays at 30s. No SSE/WS dependency.
3. **Block live refresh:** Add a `view.js` to `member-points` (and hub-tile) that re-fetches `/members/me` after `wb_gam_award_succeeded` or `wb_gam_award_skipped` postMessage events emitted by toast.js. Idempotent — page load still server-renders, JS just refreshes.

**Impact:** ~5 manifest files, toast.js, 2 new view.js files. Test for regression on async-required actions (e.g. import flows).

**Verification:** Send a kudos from frontend. Recipient's hub-tile points refresh within 10s without page reload.

### B9. `9919709692` — Merge Challenges + Community Challenges tabs (m)

**RCA:** Two separate admin pages: `wb-gam-challenges` (`ChallengeManagerPage`) and `wb-gam-community-challenges` (`CommunityChallengesPage`). Distinct top-level menu entries, distinct REST controllers (`ChallengesController` vs `CommunityChallengesController`).

**Fix:** Keep both REST controllers (tables stay distinct — different data shape). Collapse the two admin pages into one tabbed page `wb-gam-challenges` with sub-tabs "Individual" / "Community". Remove the separate menu entry; redirect old `?page=wb-gam-community-challenges` URLs to `?page=wb-gam-challenges&tab=community`. Reuse existing list/form components — they just sit under a tabset.

**Impact:** Admin menu registration (1 file removal), admin page wrapper (combine PHP), 2 list components. Backward-compat redirect for old URL. Documentation in docs/website/admin/.

**Verification:** Only one "Challenges" menu entry. Both list views work under their tabs. Old URL redirects to new.

### B10. `9919722274` — No frontend "give kudos" UI (m)

**RCA:** `KudosController.php:74` exposes `POST /kudos`. `KudosEngine` writes the row. But no block, shortcode, or modal lets a member trigger that POST from the frontend. Only programmatic / dev consumers can call the API.

**Fix:** New block `wb-gamification/give-kudos` — small button or inline form with recipient picker (BP member search if BP active, else user_login text input) + message textarea + POST to `/kudos`. Auto-include on BP member profile pages via existing ProfileIntegration hook (configurable via setting). Matching shortcode `[wb_gam_give_kudos to="user_id"]` for theme drop-ins.

**Impact:** New block (block.json, edit.js, view.js, render.php), 1 new shortcode, 1 setting toggle, 1 new admin doc page. No DB or REST changes — endpoint already exists.

**Verification:** On a BP member profile, "Send Kudos" button appears. Click → modal → fill message → submit. Recipient sees kudos in their feed within 10s (combines with B8 live refresh).

### B11. `9914952954` — Per-action frequency / cap admin UI (m)

**RCA:** `SettingsPage.php:887-892` renders `daily_cap` as a read-only pill from `$action['daily_cap']` (manifest value). Cooldown is surfaced only on the per-conversion page, not on the actions settings table. No per-action edit UI — sites must filter `wb_gam_*` server-side to change either value.

**Fix:** Make daily_cap and cooldown editable per action, persisted in `wb_gam_action_overrides` user/site option (new option key — keyed by action_id). RateLimiter reads override first, manifest as fallback. Admin row: edit-in-place pill → input → save via existing REST form driver. Reset button restores manifest default.

**Impact:** 1 new option key (no schema change), RateLimiter modified to layer overrides, settings UI edit-in-place component, REST endpoint `PUT /actions/{action_id}/overrides`.

**Verification:** Set cooldown=60s on `mvs_give_comment`. Confirm RateLimiter blocks 2nd comment award within 60s. Reset button restores manifest default (= no cooldown if manifest sets none).
