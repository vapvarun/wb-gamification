## Bucket A — Boot sequence: kill nested `plugins_loaded` pattern (1 card)

**Cards:** #9914460166 (Setup wizard not triggered on activation, three-time reopen)

**Mechanism:** `wb-gamification.php:282-307` does `add_action('plugins_loaded', SetupWizard::init, 10)` from *inside* a callback already running at `plugins_loaded` priority 0. WordPress supports this (later priorities still fire), but it makes timing fragile and obscures the actual init contract. The pattern is also a code smell: every admin module's init is registered twice (once as a hook, then dispatched again from another hook).

**Refactor:** call `::init()` directly on each admin module from `register_hooks()`. `register_hooks()` itself runs at `plugins_loaded@0`, which is before any `admin_init` listener could fire, so `SetupWizard::maybe_redirect` will always be attached to `admin_init@1` in time.

**Files:**
- `wb-gamification.php` (lines ~282-307) — strip 14 lines of nested `add_action('plugins_loaded', …)`, replace with direct calls inside `if ( is_admin() )`.

**Closes:** #9914460166. Reopen pattern caused by genuine timing sensitivity — direct dispatch removes the failure mode entirely.

**Verification:**
- Sandbox 1 — fresh install: clear `wb_gam_wizard_complete`, deactivate + reactivate plugin, visit wp-admin → expect 302 to wizard URL.
- Sandbox 2 — completed install: `wb_gam_wizard_complete=1`, visit wp-admin → expect no redirect, stays on plugin dashboard.

## Bucket B — Notification queue race condition (1 card)

**Cards:** #9927279782 (toasts not rendering for points / badges / level-up)

**Mechanism:** single transient `wb_gam_notif_{user}` is consumed-and-deleted by THREE readers in any order:
1. `NotificationBridge::render()` at footer (every logged-in page load) — `NotificationBridge.php:422`
2. `HeartbeatChannel::flush_toasts()` at heartbeat tick — `HeartbeatChannel.php:241`
3. `MembersController::get_toasts()` REST fallback — `MembersController.php:677`

Whichever fires first wins; the other two get an empty list. Plus a secondary snake_case bug: PHP queues `iconUrl`, JS reads `icon_url` (`assets/interactivity/notifications.js:94`) → level-up icon never renders even when the toast IS delivered.

**Refactor:** the notification queue must be append-only with per-consumer cursor. Replace the destructive `get_transient + delete_transient` pattern with a "delivered" set per consumer.

Design:
- Each notice gets a monotonic `event_id` (microtime-based or sequence).
- Storage: `wb_gam_notif_{user}` becomes a circular buffer of last N notices (e.g., 50) — still a transient, still self-pruning.
- Each consumer tracks its delivered cursor in user-meta: `wb_gam_notif_seen_{consumer}` (footer / heartbeat / rest). Each consumer reads notices with `event_id > cursor`, advances cursor, never deletes.
- Notice eviction handled solely by the buffer cap.

Fixes the race; allows multiple consumers to deliver the same notice once each.

Plus fix `icon_url` ↔ `iconUrl` snake_case mismatch (normalise to snake_case server-side and in JS).

**Files:**
- `src/Engine/NotificationBridge.php` — replace queue read in `render()`; expose append + read-since helpers.
- `src/Engine/HeartbeatChannel.php` — call new read-since helper, advance heartbeat cursor.
- `src/API/MembersController.php` — same in `get_toasts()`.
- `assets/interactivity/notifications.js` — read `icon_url` consistently.

**Closes:** #9927279782 (lossy delivery + icon mismatch — both root causes).

**Verification:** award points → page load → toast appears. Award badge → wait 15s → heartbeat tick toast. Award level-up → REST poll → REST toast. All three deliver same notice (no race).

## Bucket D — Block view.js hydration contract (2 cards, shared pattern)

**Cards:** #9914601059 (leaderboard badges + icons not rendering after hydration), #9919406312 (badges/streak block UI not improved — likely same enqueue/hydration cause).

**Mechanism:** Leaderboard block ships a `viewScript` that runs on every page load. `view.js:127` `buildRow()` constructs a row from heartbeat payload but emits only `rank/name/points` — no icon nodes, no badges span. `patchBoard()` (line 93) tries to match server rows by `data-user-id` but `render.php:137` doesn't emit it, so JS treats every row as new and rebuilds. The PHP-rendered icons get blown away on first heartbeat tick.

Plus secondary: `HeartbeatChannel::send_payload()` (~line 298) omits badge_count, so even if JS were patching properly, the badge span data isn't on the wire.

**Refactor:** define a "server-rendered + heartbeat-patched" contract:
1. Server render emits `data-user-id` (and `data-rank`) on every row.
2. `view.js` becomes a *patcher*, not a renderer. On heartbeat tick:
   - For each user_id in payload that has a matching server row → update only the changed text nodes (points number, rank number). Touch nothing else.
   - For new user_ids without a row → render via `buildRow()` (only path that constructs DOM from scratch).
   - For user_ids no longer present → remove row.
3. Heartbeat payload carries `badge_count` + (optionally) `points_icon_url`. JS updates the text inside the existing `.badges-count` span without recreating the span.

Apply same contract to badge-showcase + streak blocks (audit their view.js for similar destructive patterns).

**Files:**
- `src/Blocks/leaderboard/render.php` — add `data-user-id` + `data-rank` per row.
- `src/Blocks/leaderboard/view.js` — refactor `patchBoard()` to patch text-in-place; `buildRow()` only for net-new rows.
- `src/Blocks/leaderboard/block.json` — bump version if attribute schema changes.
- `src/Engine/HeartbeatChannel.php` — extend payload with `badge_count` per user.
- `src/Blocks/badge-showcase/view.js` — audit + apply same pattern.
- `src/Blocks/streak/view.js` — audit.
- `build/Blocks/…` — rebuilt via `npm run build`.

**Closes:** #9914601059, #9919406312.

**Verification:** server render → screenshot icons present. Force heartbeat tick via `wp.heartbeat.connectNow()` → icons still present, points number updated. Verify on 1440 desktop + 390 mobile. Test on member directory + leaderboard block instances.

## Bucket E — WooCommerce sync vs async routing (1 card)

**Cards:** #9925589914 (only `add_to_cart` event fires)

**Mechanism:** `integrations/woocommerce.php` registers 5 actions. 4 are `repeatable=true`. `Registry::register_action` (line 176) routes repeatable actions through `Engine::process_async` (line 119) which enqueues an Action Scheduler job. If AS workers aren't running or are backed up, events look dead. `add_to_cart` works because… unclear — possibly its own sync path elsewhere.

**Refactor:** `repeatable=true` does not need to imply async. They're orthogonal concerns:
- `repeatable` = "user can earn for this action more than once".
- `sync` = "writes to ledger inside the request, not deferred".

Order completion should be sync. The user has just paid; they expect points immediately (and if AS is broken on the site, they NEVER get them). Add explicit `sync` flag to manifest. Default false (current behaviour). Order completion gets `sync=true`.

**Files:**
- `src/Engine/Registry.php` — accept `sync` flag on register_action; route accordingly.
- `src/Engine/Engine.php` — `process_async()` becomes the conditional path.
- `integrations/woocommerce.php` — mark `wc_order_completed` + `wc_first_purchase` + `wc_product_reviewed` sync. Cart/wishlist stay async (high volume).
- `integrations/*` audit — mark the actions that obviously need sync delivery.

**Closes:** #9925589914.

**Verification:** test order in WC → ledger row exists immediately after thank-you page, not after AS worker tick.

## Bucket L — Cron fatal: environmental (1 card)

**Cards:** #9927883656 (wp cron event run wb_gam_cohort_assign fatal: undefined function mysqli_report)

**Mechanism:** Error is in `wp-includes/class-wpdb.php:1964` calling `mysqli_report()`. That function exists in PHP if and only if the `mysqli` extension is loaded with `mysqli.allow_persistent` etc. configured. On a fresh PHP install without mysqli, this fails.

This is **not a wb-gamification bug** — WP core fails to bootstrap before any plugin code runs. The cron handler hasn't even been entered.

**Approach:** reproduce locally with `wp cron event run wb_gam_cohort_assign`. If it works here but not on Simran's box, file as an environmental issue (PHP mysqli extension missing on her local). If reproducible, escalate to WP-Cli / wpdb upstream.

**Closes:** #9927883656 if confirmed environmental.
