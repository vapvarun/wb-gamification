  `/wp-plugin-onboard --refresh` should pick the tables/cron up; if it does not, the generator

  not a forced new feature.
- **PHPStan caveat.** Project standard is level 9, but PHPStan silently exits 0 in
  Local-by-Flywheel PHP (memory: `reference_phpstan_broken_locally`). The authoritative
  PHPStan gate is **GitHub CI on the PR**, not the local static gate. All work lands via PR
  on protected `main` (memory: `feedback_use_prs_not_admin_bypass`).
- **ApiKeyAuth unbounded query (audit finding #9), wc double-hook (#11), doctor false
  positives (#12), hooks_fired drift (#10), REST 0% coverage (#7)** are **out of scope** for

## Wave ordering — why this sequence

The proven 9-phase order (`data-layer → … → testing → build-deploy`) collapses here because
the three code fixes have **non-overlapping file scope** and no inter-dependencies, so they
run as one parallel wave:

1. **Wave 1 (parallel code fixes)** — 1.1 badge, 1.2 a11y, 1.3 WC endpoint. Disjoint scopes
   (integration manifest + installer; admin views + CSS; WC integration). All `fix` journey.
   This is the actual customer-facing repair work and must land before the manifest is
   re-generated.
2. **Wave 2 (manifest drift)** — depends on Wave 1 so the regenerated manifest also captures
   the badge/endpoint deltas. 2.1 (tables + three-entry-point) and 2.2 (cron + smoke coverage)
   have disjoint scope (`manifest.json`/entry-point files vs `qa-coverage.json`/cron-engine
   files + `tests/`). Investigate the **shared generator root cause first**, then split.
3. **Wave 3 (audit close)** — read-only `audit` journey; re-runs release/04, /07, /15 and the
   drift check and writes the SHIPPABLE verdict. Depends on both prior waves.

Same-wave scopes are disjoint (validator-enforced) so the master can run each wave's
directions in parallel worktrees.

## Root-cause hypotheses per finding

### #1 — Badge auto-award broken on BuddyPress sites (MAJOR → direction 1.1)
`integrations/wordpress.php:125` marks `wp_publish_post` (and the comment action) `standalone_only: true`.
`ManifestLoader` (src/Engine/ManifestLoader.php:235) strips `standalone_only` actions when
BuddyPress is active, on the theory "BP covers the same event." But the 5 seeded badges
(`Installer.php:709-733`) bind their conditions to those exact WP action IDs — and there is **no
BP-equivalent for publishing a `post` or leaving a WP `comment`** (BP activity updates and
activity comments are different events). So on the primary deployment target the actions never
load and the badges never fire; `wp wb-gamification doctor` already warns about it.

**Hypothesis:** the `standalone_only: true` flag is *wrong* for `wp_publish_post` /
`wp_leave_comment` — these are genuine WP-core events with no BP analog, unlike (say) a
profile update. **Likely fix:** drop `standalone_only` for these two so they stay registered
alongside BP (the installer comment at line 707 already *asserts* they are "unconditionally
registered… no BuddyPress required," so the flag contradicts the documented intent). The
alternative — remapping badge conditions to `bp_*` actions — is wrong because publishing a
blog post is not a BP activity. The fix must make `doctor` go quiet and keep the action IDs
unchanged (contract).

### #2 — 3 live tables undocumented in manifest (MAJOR → direction 2.1)
`wb_gam_notifications_queue`, `wb_gam_side_effect_failures`, `wb_gam_user_intelligence` are
created by `DbUpgrader::ensure_*_table()` (lines 97-99) as **idempotent feature-migrations on
existing installs**, NOT in `Installer.php`'s primary `CREATE TABLE` block.

**Hypothesis:** the manifest generator enumerates tables from `Installer.php` (the canonical
schema) and does not walk `DbUpgrader`'s `ensure_*` migration methods — so any table added
*only* as a later migration is invisible to the scan. Fix = (a) teach the generator to read
`DbUpgrader` migration tables, and (b) add the three-entry-point note per table
(`user_intelligence` already has `IntelligenceController` REST; `side_effect_failures` is
internal engine accounting and likely an intentional exception; `notifications_queue` is fed
by the SSE/Heartbeat path).

### #3 — ~3 cron hooks missing from manifest (MAJOR → direction 2.2)
Same generator-gap class as #2. Confirmed live: the manifest cron[] already lists 11 (incl.
`wb_gam_as_cleanup`, `wb_gam_points_decay`). The genuine miss is the **3 crons registered
inside engine classes** — `wb_gam_reconcile_side_effects` (SideEffectDispatcher.php:63, hourly),
`wb_gam_compute_intelligence` (IntelligenceProjector.php:73, daily),
`wb_gam_notifications_queue_prune` (NotificationBridge.php:148, daily). (The audit's "4th"
candidate `wb_gam_as_cleanup` is already in the manifest — no action needed there.)

**Hypothesis:** the generator detects crons via `CRON_HOOK`-named constants / a known schedule
pattern and missed these three because they use differently-named constants
(`RECONCILE_CRON` / `COMPUTE_CRON` / `PRUNE_CRON`) on `wp_schedule_event`. `qa-coverage.json`
showing 9 is a downstream symptom of the same stale enumeration. Fix = correct the generator's
cron detection, register all three in the manifest, add smoke-level cron coverage.

### #4 — 3 WCAG violations (MINOR → direction 1.2)
Baseline-known since 2026-05-27, never closed. (a) `<img>` without `alt` at
`BadgeAdminPage.php:285,516`, `SubmissionsPage.php:197`, `templates/emails/badge-earned.php:53`;
(b) `outline:none` without `:focus-visible` in 5+ selectors across `assets/css/admin*`,
`assets/css/frontend.css`, `src/Blocks/submit-achievement/style.css`; (c) 2 unlabeled inputs at
`PointTypeConversionsPage.php:247,259`.

**Hypothesis:** straightforward omissions, not a systemic gap — fix per `ux-foundation` rules
(non-empty alt or `alt=""` for decorative; `:focus-visible` ring replacement for every
`outline:none`; `<label for>` / wrapping label for the two inputs). The `ux-audit` static gate
will confirm closure.

### #5 — /my-account/achievements/ 404 (MINOR → direction 1.3)
`AccountIntegration.php:61-63` flushes rewrite rules exactly once, guarded by the option
`wb_gam_wc_account_endpoint_v1`. On installs where that option was set in a build *before* the
endpoint existed (or before WooCommerce's account router knew the query var), the guard is
already "spent," so the one-time flush never re-runs and the endpoint stays unresolved.

**Hypothesis:** classic one-shot-flush-fired-too-early ordering bug. **Fix:** make the guard
**version-aware** (bump the option key, e.g. `_v2`, or store `WB_GAM_VERSION` and re-flush when
it changes) or **self-healing** (detect that the endpoint's rewrite rule is absent from the
stored rules and flush). Reproduce the 404 live first to confirm the stored-rules state before
choosing the guard strategy.

  the PK-bearing columns I could confirm by name (`id`, `user_id`, `payload` / `action_id`).
  Wave 2.1 should fill the complete column set when it documents them in the manifest.

  cleanly from `src/CLI/` in this pass (likely registered via a loader/array). I listed the 4
  the audit smoke-verified (`doctor`, `points award`, `member status`, `actions list`) as the
  contract floor; this is not the full set and should not be used to prune commands.
