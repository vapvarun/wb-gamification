## What's still flagged (and why each is acceptable)

### 1. 2× a11y `outline:none without :focus-visible` — heuristic mismatch

The audit flags 2 outline:none findings but **all 4 source-CSS sites have a paired `:focus-visible` rule with an explicit `outline: 2px solid` declaration**:

| File:line of `outline: none` | Paired `:focus-visible` rule |
|---|---|
| `assets/css/admin.css:431` (`.wbgam-input:focus`) | `assets/css/admin.css:436-439` |
| `assets/css/admin.css:451` (`.wbgam-select:focus`) | `assets/css/admin.css:456-459` |
| `assets/css/admin-premium.css:529` (`.wbgam-input`) | `:543-546` |
| `assets/css/admin-premium.css:575` (`.wbgam-textarea`) | `:585-588` |
| `assets/css/admin-premium.css:603` (`.wbgam-select`) | `:619-622` |
| `assets/css/admin-premium.css:688` (`.wbgam-btn`) | `:691-695` (added in PR #9) |

The audit's heuristic doesn't recognize these as paired. Possible causes: looking for a specific selector-chain match, or scanning a transformation of the CSS. Manual a11y verification (keyboard-tab through every form, button, and select) confirms a visible focus indicator on every interactive element. Acceptable.

### 2. 4× admin-eval `Direct $_POST to update_option` — intentional architecture

The Settings page (and 3 other admin pages) handle their own form submission via direct `$_POST` reads + `update_option()` calls instead of the WordPress Settings API. The audit recommends `register_setting()` + `settings_fields()`.

This is a deliberate architectural choice — see `src/Admin/SettingsPage.php` and the team's decision to use a custom card-based admin UI rather than the legacy `<table>` Settings API rendering. Migrating would be a substantial refactor across 4 admin pages and provide no functional benefit (the same nonce/cap checks already gate every write). Not in scope.

### 3. 4× critical "fix before release" GAP findings — all verified false positives

| Audit claim | Reality |
|---|---|
| "Some REST routes lack proper permission callbacks" | All 39 routes have explicit `permission_callback`. The 17 `__return_true` ones are intentionally public (catalog reads, OG share, OpenBadges credential) per `audit/ROLE_MATRIX.md`. |
| "Custom tables exist but no uninstall hook" | `uninstall.php` exists; full cleanup. |
| "Custom tables exist but no activation hook" | `register_activation_hook` at `wb-gamification.php:320`. |
| "Cron jobs scheduled but no deactivation hook" | `register_deactivation_hook` at `wb-gamification.php:337`. |

These were verified one-by-one in PR #2 and remain false positives — the audit's heuristic doesn't recognize the existing implementations. Each new audit run will re-flag them.

### 4. PHPCS error count — wppqa's ruleset, not the project's

The `734 errors / 565 warnings` count is wppqa's own stricter PHPCS ruleset (which doesn't read the project's `.phpcs.xml`). The project's actual `composer phpcs` against `.phpcs.xml` returns clean — verified by the GitHub Actions \"WordPress Coding Standards\" check passing on every PR in this campaign. See PR #3 SUMMARY for the full explanation.

## Optional follow-ups (not blockers)

These remain for the team to schedule when the time is right:

1. **Migrate admin pages to Settings API** — closes the 4 `admin-eval` audit findings. Major refactor. Defer until there's a UX redesign opportunity.
2. **Convert remaining admin pages to granular caps** — `SettingsPage`, `SetupWizard`, `ApiKeysPage` could each get their own granular cap (`wb_gam_manage_settings`, `wb_gam_run_setup`, `wb_gam_manage_api_keys`). Today they keep `manage_options` for security/simplicity.
3. **Investigate the 2 persistent a11y outline:none flags** — not actionable without seeing the audit's exact heuristic. Could request a wppqa upgrade that documents the rule, or accept the heuristic mismatch.
4. **Remove the 28-pt and 1-event drift from the journey baseline run** — the dev DB has slightly elevated state from journeys. Production runs would teardown.
5. **BuddyPress profile screenshot** — `screenshot-9` was originally meant to be a BP profile but BP is not active on this Local. Capture against a BP-active site when one is available.
